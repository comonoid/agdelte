# Концепция хранения: обычные типы + индексы

Ментальная модель доменного/хранилищного слоя CRM. Companion к
[ADR 0001](../adr/0001-storage-wal-in-memory.md) (решение и обоснование) — здесь
*как об этом думать*.

## Идея в одной фразе

> **Предметную область описываем обычными типами + индексами — ровно так, как
> писали бы при обычной работе.** Обычные типы слегка обосабливаются, чтобы их
> изменения можно было писать на диск как WAL. Индексы на диск не пишутся —
> живут только в памяти и восстанавливаются проигрыванием лога.

Никакого ORM, schema-first или чужой модели данных, чьим неестественным «view»
была бы программа. Типы Agda **и есть** модель.

## Зачем так: не ломать интуицию обычного ФП

**Это сделано ради одного — чтобы не ломать интуицию.** С этими (персистентными,
журналируемыми) типами обращаешься **почти точно так же, как пишешь обычную
программу на функциональном языке**: определяешь записи и суммы, пишешь над ними
**чистые функции** (`apply`, запросы), компонуешь — как всегда. Новой парадигмы
учить не надо.

«Обособление» (изменения через `Op` + сериализация + id-ключи) — это **минимальная
дельта**, а не переход в другой мир: нет SQL, нет ORM-маппингов, нет языка
запросов, нет церемонии транзакционного API. Персистентность и транзакционность —
**почти бесплатная добавка** к обычному ФП-коду.

Контраст: ORM/SQL заставляет держать **две** ментальные модели (объекты программы
и реляционную схему), переводить между ними и думать запросами на чужом языке —
вот это **ломает** интуицию. Здесь — **одна модель, одна парадигма, нулевой
paradigm-tax.** Это и есть глубинная причина, по которой подход стоит того: не
«быстрее/дешевле», а **думаешь как обычно**.

## 1. Обычные типы — «база» (источник истины)

Доменные сущности — обычные записи/суммы/вложенные значения, как в любой
программе: `Party`, `Engagement`, `Activity`, а M:N — это запись-факт
`Participation { engagementId; partyId; role }`. Владеемое (адрес, attrs) —
вложено значением; идентичность/разделение — через id (см. ADR: граница агрегата).

## 2. «Обособление»: чтобы изменения шли в WAL

Минимальная церемония, превращающая обычные типы в журналируемые:

1. **Изменения — через явные операции** (`Op`-сумма: `CreateParty`, `RenameParty`,
   `AddParticipant`, …), а не произвольная мутация на месте. Операция — это то,
   что пишется в лог.
2. **База сериализуема** (`Serialize` для `Op`) — одна строка лога на операцию.
3. **Сущности идентифицированы** (id-ключ в `NatMap`), чтобы операция адресовала
   «кого менять».

Всё остальное — обычные значения. `apply : Op → Base → Base` — чистая функция.

## 3. Индексы — производные, в памяти, НЕ пишутся

Индексы (обратные/вторичные, бидирекциональные `Rel` для M:N) нужны только для
эффективных запросов. Они — **чистая функция от базы**:

```
reindex : Base → Indexes
```

Поэтому:
- **на диск не попадают** (WAL хранит только операции над базой);
- **восстанавливаются при старте** — реплей операций перестраивает и базу, и
  индексы; индексы существуют только в RAM.

Persistence-слой видит **только** простые доменные записи; индексы — невидимая для
диска оптимизация. Сменил представление индекса — формат лога не изменился.

## 4. Обязательство консистентности + страховка

Цена: индексы надо держать согласованными с базой на **каждом** изменении —
сейчас вручную в `apply`, и тут **легко забыть обновить**. Страховка — то, что
индексы выводимы:

> **Инвариант `idx ≡ reindex base`** — проверяемый. Инкрементальная поддержка в
> `apply` обязана его держать; покрывается **property-тестом** (после произвольной
> последовательности операций `idx` совпадает с пересчётом с нуля).

Это снимает главный риск ручной поддержки — «забыл обновить индекс» ловится тестом.

## 5. Куда расти (потом)

Сейчас связь индекс↔данные держится руками в `apply`. Будущее направление —
**декларативно задавать отношение индекса к данным**, чтобы он обновлялся
**автоматически** (incremental materialized view: объявил `byParty = index
participations partyId` — движок сам поддерживает на insert/update/delete). Это
вернуло бы полную прозрачность («не думать об индексах»), но машинерию строить
только когда реляций станет много — при горстке дешевле держать руками.

## Маленький пример

```
-- база (пишется в WAL как операции):
record Party         field id; type; name
record Participation field engagementId; partyId; role

record Base    field parties : NatMap Party ; participations : NatMap Participation ; …

-- индекс (производный, в памяти, не пишется):
record Indexes field engByParty : NatMap (List ℕ)   -- partyId → [engagementId]

reindex : Base → Indexes        -- свёртка participations в обе стороны
-- старт: replay операций → base ; затем idx = reindex base
-- инвариант: после apply  idx ≡ reindex base   (property-тест)
```

---

Суть: **домен — обычные типы (плюс M:N как запись-факт); индексы — производный
in-memory слой, не сериализуется, держится инвариантом `idx ≡ reindex base`.**
Persistence знает только про простые записи. Прозрачно как обычное
программирование, плюс журнал изменений и восстановление реплеем.
