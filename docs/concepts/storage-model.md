# Концепция хранения: типы + индексы + операции + транзакции

Полная ментальная и техническая модель доменного/хранилищного слоя CRM. Companion к
[ADR 0001](../adr/0001-storage-wal-in-memory.md) (решение и обоснование) — здесь
*как это устроено и как об этом думать*.

Типы в примерах — иллюстративный псевдо-Agda; имена черновые.

---

## Идея в одной фразе

> **Предметную область описываем обычными типами + индексами — ровно так, как
> писали бы при обычной работе.** Обычные типы слегка обосабливаются, чтобы их
> изменения можно было писать на диск как WAL. Индексы на диск не пишутся —
> живут только в памяти и восстанавливаются проигрыванием лога.

Никакого ORM, schema-first или чужой модели данных, чьим неестественным «view»
была бы программа. Типы Agda **и есть** модель.

## Зачем так: не ломать интуицию обычного ФП

**Сделано ради одного — чтобы не ломать интуицию.** С этими (персистентными,
журналируемыми) типами обращаешься **почти точно так же, как пишешь обычную
программу на функциональном языке**: определяешь записи и суммы, пишешь над ними
**чистые функции**, компонуешь — как всегда. Новой парадигмы учить не надо.

«Обособление» — **минимальная дельта**, а не переход в другой мир: нет SQL, нет
ORM-маппингов, нет языка запросов. Персистентность, индексы и транзакционность —
**почти бесплатная добавка** к обычному ФП-коду. Контраст: ORM/SQL держит **две**
модели (объекты программы и реляционную схему) и переводит между ними — вот это
ломает интуицию. Здесь — **одна модель, одна парадигма, нулевой paradigm-tax.**

Это критерий и на будущее: **любая фича, которая ломает «обращаешься как с
обычными типами», — подозрительна.**

# Часть I — Данные

## 1. База — обычные типы (источник истины)

Доменные сущности — обычные записи/суммы/вложенные значения:

```
data PartyType = person | org
record Party         { id : ℕ ; type : PartyType ; name : String ; tz : String }
record Engagement    { id : ℕ ; caseType : ℕ ; stage : ℕ ; title : Maybe String }
record Activity      { id : ℕ ; engagementId : ℕ ; startsAt : ℕ ; status : ActStatus }  -- 1:N: FK-поле
record Participation { engagementId : ℕ ; partyId : ℕ ; role : Role }                    -- M:N как запись-факт
```

**База** — набор коллекций по типам:

```
record Base
  { parties        : Table Party
  ; engagements    : Table Engagement
  ; activities     : Table Activity
  ; participations : Table Participation
  ; nextId         : ℕ }
```

## 2. «Столбцы» держат любые индуктивные типы

Главное отличие от SQL: **поле сущности — любой тип Agda**, first-class,
типизированный, с pattern-matching: рекурсивные ADT, суммы, вложенные записи,
деревья. Поэтому **владеемое не нормализуется** в отдельные таблицы:

```
data Tree a = leaf a | node (Tree a) (Tree a)
record Category { id : ℕ ; tree : Tree Label }   -- дерево лежит ДЕРЕВОМ, не adjacency-list
```

- **навигация/глубокое обновление** — оптики (`Reactive/Optic`, `BigLens`) / зипперы:
  `over path f tree` → новое дерево со structural sharing;
- **граница агрегата:** идентичность/разделение/M:N → ссылка по id; владение/нет
  разделения/поддерево → вложенное значение. Уплощение в таблицу **только** когда
  узлы обретают самостоятельную идентичность (тогда они — сущности).

Это ровно то, чего лишён SQL (плоские скаляры + JSONB-escape-hatch); см. §12.

## 3. Таблицы — `IndexedMap` (абстрактный тип с авто-индексами)

`Table` = `IndexedMap K V`: коллекция по первичному id + **объявленные** вторичные
индексы, которые `insert`/`update`/`delete` поддерживают **сами**. Реализация —
постулат над хаскельной структурой (как `NatMap`): `IntMap V` + по `IntMap`-у на
каждый индекс. Представление **скрыто** — только smart-конструкторы.

```
postulate IndexedMap : Set → Set
insert  : ℕ → V → IndexedMap V → IndexedMap V        -- правит ВСЕ индексы
delete  : ℕ → IndexedMap V → IndexedMap V
lookup  : ℕ → IndexedMap V → Maybe V
byIndex : IndexName → ℕ → IndexedMap V → List ℕ      -- доступ по вторичному ключу
```

Индекс объявляется экстрактором `V → List ℕ` (для дерева — фолд). **Почему это
идиоматично:** `Data.Map`/`IntSet`/`FingerTree` так же держат внутренний инвариант
(балансировка / моноидальная мера) за абстракцией со smart-конструкторами;
`IndexedMap` держит инвариант **согласованности индекса**. Инкапсуляция (через
постулат-над-Haskell — бесплатно) делает несогласованное состояние **невыразимым
снаружи**. Снаружи это «умная карта», а не DSL → интуиция цела.

**Связи через `IndexedMap`:**
- **1:N** (engagement→activities): таблица `Activity` с индексом по `engagementId`.
- **M:N** (party↔engagement): таблица `Participation` (запись-факт) с **двумя**
  индексами — по `engagementId` и по `partyId`. **Оба** выводятся из **одной**
  коллекции → поддержка «внутри одного типа». **Кросс-типовой индекс-машинерии не
  нужно вообще** (см. §11 про каскады/агрегаты).

## 4. Индексы производны и НЕ сериализуются; in-memory ≠ wire

Индексы — **чистая функция от базы** (`reindex : Base → Indexes`), нужны только для
запросов:
- **на диск не попадают** (WAL хранит только операции над базой);
- **восстанавливаются** реплеем (или `reindex` после загрузки).

Тот же закон — **на уровне значения**: храним **эффективную структуру** в памяти,
сериализуем **логическое содержимое**. `Map Int String` лежит реальной картой, а в
лог едет **списком пар** (в каноническом порядке по ключу → детерминированные байты
+ сборка за O(N) через `fromAscList`). Внутреннюю раскладку дерева карты не пишем —
восстанавливаем при `decode`. Верхнеуровневые таблицы вообще не сериализуются
целиком (реплей их строит); «список пар» применяется к map-значным **полям** внутри
сущности.

> **Сериализуем СОДЕРЖИМОЕ, а не СТРУКТУРУ** — на всех уровнях (база/индексы,
> содержимое/раскладка). Поэтому смена in-memory представления (или индекса) **не
> трогает формат лога**.

## 5. Инвариант `idx ≡ reindex base`

Цена производных индексов: их надо держать согласованными на каждом изменении.
Страховка — выводимость:

> **`idx ≡ reindex base`** — проверяемый инвариант. Авто-поддержка `IndexedMap`
> держит его **по построению**; дополнительно **property-тест** (после произвольной
> последовательности операций `idx` совпадает с пересчётом с нуля).

# Часть II — Изменения

## 6. Операции (`Op`): реификация изменений

Чтобы изменение **журналировалось**, его выражают **сериализуемой командой** `Op` +
интерпретатором `apply` + `Serialize`. Это узнаваемая ФП-идиома «effects/commands as
data» (родня free-монадам) — не анти-ФП, а конкретный её стиль.

```
data Op = SetEntity EntityTag ℕ EntityValue   -- generic upsert — покрывает почти весь CRUD
        | DeleteEntity EntityTag ℕ
        | AddParticipant ℕ ℕ Role              -- доменные команды
        | …
apply : Op → Base → Base                       -- чистый интерпретатор (через IndexedMap.insert/delete)
```

**Гранулярность — по умолчанию upsert целой сущности.** Тогда тип обязан реализовать
лишь `Serialize` (рекурсивный/выводимый, включая вложенные деревья) + экстракторы
индексов. Цена: в лог едет всё значение сущности на изменение (для CRM-размерных —
норма). **Опционально** для крупных горячих вложенных структур — тонкие
path-адресованные edit-операции (больше конструкторов `Op`, меньше лог). Это
opt-in оптимизация, не усложнение модели.

> Интерфейс типа = **`Serialize` (всегда) + экстракторы индексов (если индексируется)
> + [опц.] словарь edit-операций (только крупные горячие вложенные структуры).**

## 7. Монада `Txn` — «упаковка» (прячет реификацию)

Чтобы тело транзакции читалось как **прямые правки**, а не ручная сборка `Op`,
заворачиваем в конкретную монаду (State+Writer+Either, **без** Free):

```
Txn A = Base → Either Err (Base × List Op × A)

return a   = λ b → inj₂ (b , [] , a)
m >>= f    = λ b → case m b of inj₁ e            → inj₁ e
                               inj₂ (b' , os , a) → case f a b' of
                                  inj₁ e             → inj₁ e
                                  inj₂ (b'' , os' , c) → inj₂ (b'' , os ++ os' , c)

getBase : Txn Base ;  getBase = λ b → inj₂ (b , [] , b)
abort   : Err → Txn A ; abort e = λ _ → inj₁ e
emit    : Op → Txn ⊤
emit op = λ b → inj₂ (apply op b , op ∷ [] , tt)    -- ОБНОВЛЯЕТ рабочее состояние тем же apply
```

Ключ — `emit`: он **применяет операцию к рабочему состоянию И логирует её**. Поэтому:
- **read-your-writes даром** — следующее чтение видит применённое;
- **`live ≡ replay` по построению** — рабочее состояние = бегущая свёртка `apply` по
  тем же `Op`, что уйдут в лог = ровно то, что даст реплей.

Доменные «сеттеры» определяются один раз поверх `emit`:

```
setName : ℕ → String → Txn ⊤
setName pid x = emit (SetEntity PartyTag pid (renamed pid x))   -- или доменная команда
```

И тело транзакции читается натурально:

```
bookSession : ℕ → ℕ → Txn ActId
bookSession eng slot = getBase >>= λ b →
  if slotTaken slot b then abort Conflict
  else newId >>= λ aid → emit (SetEntity ActivityTag aid (activity aid eng slot)) >> return aid
```

**Монада не убирает налог `Op`/`apply`/`Serialize` — переносит его.** Словарь команд
+ их семантика/сериализация определяются **один раз** в примитивах; call-site'ы
читаются как обычный код.

## 8. Applicative — когда и зачем

Дихотомия: **Монада** (`m >>= f`) — продолжение спрятано за функцией `f`, поэтому
программу в общем случае **нельзя статически осмотреть/сериализовать**, зато есть
data-dependent ветвление (нужное нам: «если слот занят…»). **Applicative**
(`Free Applicative`) — структура статична → программу можно **обойти не запуская** →
**сериализовать/анализировать/батчить**.

**Что выбираем мы:** нам **не нужно** сериализовать саму программу-транзакцию — мы
сериализуем её **результат**. Транзакция монадична; мы **запускаем её против текущего
`Base`** → все runtime-решения резолвятся → получаем **конкретный `List Op`** уже
определённых значений → журналируем **их**. Поэтому **конкретной монады `Txn`
достаточно**; статическая сериализуемость для `Op` не нужна (после прогона ops —
конкретные значения).

**Applicative держим в резерве** — он понадобится, только если захотим
сериализовать/анализировать саму **программу** транзакции (отправить на удалённый
узел, «подготовить» как prepared statement, оптимизировать до запуска). Мы реплеим
**ops, а не программы** → пока не нужно.

# Часть III — Транзакции

## 9. `walTxn`: single-writer, серийно, serializable даром

Транзакция исполняется атомарно под единым замком (`MVar` состояния):

```
walTxn : WalHandle → Txn A → IO (Either Err A)
-- takeMVar base → run txn → case:
--   inj₂ (base' , ops , a) → аппенд ops в WAL → putMVar base' → return (inj₂ a)
--   inj₁ e                 → putMVar base (как был) → return (inj₁ e)   -- откат = no-op
```

- **один писатель → serializable** тривиально (нет конкурентной записи);
- **откат — no-op** (иммутабельность: просто не ставим новое значение; ни undo, ни
  мусора, ни VACUUM);
- **чтения** — `readMVar` (консистентный снимок, конкурентно, не блокируя писателей);
- **интерактивные/оптимистичные** правки — отдельный паттерн validate-and-commit
  поверх `walTxn` (перечитать предусловия на коммите);
- **конкуренция/изоляция (SSI, двойной индекс транзакций)** — **паркуется** (см. ADR;
  это фронтир, не для текущей ниши).

## 10. Ошибки `Err` и разрешимость — нужна ли она тут?

`Err` — типизированная сумма, она же конверт ошибок headless-API (§13 SPEC):

```
data Err = NotFound | Conflict | Insufficient | InvalidTransition | Forbidden | Invariant String
```

**Транзакция фаллибельна по типу** (`Either Err`). Отказ из-за неподходящих данных
(«баланс мал», «слот занят») — **неустранимый рантайм-факт**, доказательством не
записывается. Это нормально и правильно.

**Разрешимость (`Dec P`) — да, но точечно, для value-инвариантов.** Она —
**мост** между runtime-сравнением и доказательством: `_≤?_`/`_≟_`/decidable-членство
в enum-множестве **запускается в рантайме** и в success-ветке **отдаёт
доказательство**, которое течёт в evidence-требующую операцию (correct-by-construction):

```
charge : (amt : ℕ) → ℕ → Txn ⊤
charge amt pid = getBase >>= λ b → case lookupBalance pid b of
  nothing  → abort NotFound
  just bal → case amt ≤? bal of
    no  _ → abort Insufficient
    yes p → emit (Charge pid amt {- декремент требует p : amt ≤ bal -})
```

Гарантия — **не «нет отказа», а «нет непроверенного коммита»**: к `Charge` нельзя
прийти, не предъявив `p`, а добыть `p` можно только через `≤?` + `yes`. Coverage-
checker заставляет покрыть и `no` (отказ нельзя молча пропустить).

**Где разрешимость НЕ применяется:** store-инварианты (FK, уникальность) — над
постулированным `IndexedMap` Agda **не доказуема** (структура непрозрачна); там
**рантайм-проверка** (`lookup` → `just`/`nothing`) + **property-тест**. Итог:
- **value-инварианты** (баланс ≥ 0, валидный переход стадии) → correct-by-construction
  через разрешимость, **где дёшево**;
- **store-инварианты** (FK/уник) → рантайм + property-тест;
- **тяжёлые мета-теоремы сохранения** → property-тест (не доказывать).

## 11. Кросс-типовое: каскады и агрегаты — это транзакции, не индексы

- **Каскады/целостность** («удалили party → удалить participations») — **логика
  транзакции** (эмитит операции в несколько коллекций); индексы каждой коллекции
  обновляются внутри своего типа. Не индекс-машинерия (и в SQL — тоже логика).
- **Материализованные кросс-коллекционные агрегаты** (денормализованный join/счётчик)
  — единственное по-настоящему кросс-типовое; **опционально и редко** → паркуется.

## 12. Внешние эффекты — через outbox

Транзакция чиста → внешний эффект (списать карту, послать вебхук) в неё не втянуть.
Паттерн (SPEC §6): транзакция **атомарно пишет намерение** (операция → запись
outbox), отдельный исполнитель выполняет эффект **at-least-once + идемпотентно** (по
`provider_ref`). Транзакции + outbox = денежная/коммуникационная история.

# Часть IV — Рамка

## 13. Отношение к SQL и prior art

Мы сошлись на **реляционном ядре** (таблицы, индексы, связи, транзакции) — потому
что оно фундаментально, а **не** «переизобретаем SQL по бедности». Расходимся там,
где SQL слаб — **в типах**: «столбцы» держат **любые индуктивные типы**, всё в одном
зависимо-типизированном языке, инварианты проверяет компилятор. Это:
- **Date & Darwen, «The Third Manifesto»** — «реляционная модель верна, SQL — кривая
  реализация, типы должны быть полноценными» (язык вместо SQL). Наш случай.
- **`acid-state` (Haskell)** — ближайшая prior art: in-memory типизированное
  состояние + журнал событий + транзакции-как-чистые-функции, без SQL.

Отдаём SQL: **ad-hoc декларативные запросы + планировщик**, конкурентность, масштаб
за RAM. Поэтому «как SQL» — про модель; **лучше SQL** — для ниши «маленький
типизированный домен» (нет impedance, настоящие типы, транзакции даром, durability).

## 14. Инварианты-страховки (сводка)

- **`idx ≡ reindex base`** — индексы выводимы (Часть I).
- **`live ≡ replay`** — рабочее состояние = свёртка `apply` по ops, держится `emit` (Часть II).
- **value-инварианты** — correct-by-construction через разрешимость, где дёшево (§10).

## 15. Что отложено (parked)

Снапшоты (WAL-only) · пейджинг/list-of-pages · конкуренция/изоляция (SSI, двойной
индекс транзакций) · `Free Applicative` для статически-сериализуемых программ ·
тяжёлые correct-by-construction-доказательства · материализованные кросс-коллекционные
агрегаты · декларативный auto-index DSL · Postgres (scale-out за headless-API).

# Часть V — Как это работает (рантайм)

## 16. Раскладка по модулям

- **`agdelte`** (генерик-инфра): `Storage.WAL` (`walOpen`/`walStep`/`walRead`/**`walTxn`**),
  `Storage.NatMap`, **`Storage.IndexedMap`** (новый), `FFI.{Server,FileSystem}`.
- **`services-core`** (CRM-домен): `Crm.Identity` (записи сущностей), **`Crm.Store`**
  (`Base`/`Indexes`/`CrmOp`/`apply`/`reindex`/`Serialize`/`Err`/`WalConfig`),
  **`Crm.Txn`** (монада `Txn` + `emit` + доменные команды), **`Crm.Api`**
  (HTTP-обработчики → `walTxn`/`walRead`).

## 17. Что лежит в памяти

Единое `CrmState = Base × Indexes` в **одном `MVar`** (единственный писательский
замок). `Base` — таблицы (`IndexedMap` по сущностям); вторичные индексы — внутри
`IndexedMap`. Всё иммутабельно; новая версия делит структуру со старой.

## 18. Формат WAL-файла

Один `Op` = одна запись (length-prefixed — устойчиво к рваному хвосту; `fsync` на
аппенд). Только операции над базой; индексов и снапшотов на диске нет. Схематично:

```
set|party|7|person|Полунин|Europe/Moscow
addpart|3|7|provider
```

## 19. Старт (recovery)

1. `walOpen`: `emptyBase` → читаем WAL → для каждой записи `deserialize → apply` →
   база восстановлена.
2. `reindex base` → индексы (или их строит тот же `apply` по ходу реплея).
3. `state := Base × Indexes` в `MVar`.

Рваный хвост последней записи пропускается; незакоммиченного частичного состояния
нет (аппенд — **только на коммит**), поэтому реплей даёт ровно докрашевое состояние.

## 20. Жизненный цикл ЗАПИСИ (end-to-end)

HTTP `POST` → обработчик парсит тело → строит `Txn A` (доменная команда) → `walTxn h txn`:

1. `takeMVar state`;
2. прогнать `txn` против `base` → `Either Err (base', ops, a)`;
3. `Right`: аппенд каждой `op` в WAL (`serialize`+`fsync`) → `putMVar state'` → вернуть `a`;
4. `Left e`: `putMVar state` (без изменений) → вернуть `e` (**откат = no-op**);

→ обработчик формирует ответ (`{data}` / `{error}` по SPEC §13).

## 21. Жизненный цикл ЧТЕНИЯ

HTTP `GET` → обработчик → `readMVar state` → **чистый** запрос (`lookup`/`byIndex`) →
сериализовать результат → ответ. Без WAL, без блокировки писателей.

## 22. Связь движка с доменом (`WalConfig`)

`walApply := Crm.apply` · `walSerializeOp`/`walDeserializeOp := CrmOp`-кодеки ·
`walEmptyState := emptyBase` · снапшотные поля — заглушки (WAL-only). Движок
(`agdelte`) ничего не знает про домен; домен (`services-core`) поставляет конфиг.

## 23. Конкурентность рантайма

Записи **сериализованы** `MVar`-ом (один `walTxn` за раз → serializable даром);
чтения через `readMVar` **конкурентны** и не блокируют писателей. Warp даёт поток на
запрос; каждый идёт через движок. (Конкурентная запись/SSI — паркуется, §15.)

## Пример целиком (минимальный срез)

```
-- ДАННЫЕ
record Party         { id ; type ; name }
record Participation { engagementId ; partyId ; role }
record Base    { parties : Table Party ; participations : Table Participation ; nextId : ℕ }
-- индекс по partyId объявлен на таблице participations → byIndex даёт engagement'ы party

-- ОПЕРАЦИИ
data Op = SetEntity Tag ℕ Value | AddParticipant ℕ ℕ Role | …
apply : Op → Base → Base                              -- через Table.insert/delete

-- ТРАНЗАКЦИЯ
addParticipant : ℕ → ℕ → Role → Txn ⊤
addParticipant eng p role = getBase >>= λ b →
  case lookup p (parties b) of
    nothing → abort NotFound                          -- store-инвариант: рантайм
    just _  → emit (AddParticipant eng p role)        -- emit: apply + лог; live≡replay

-- ИСПОЛНЕНИЕ
result ← walTxn h (addParticipant eng p role)         -- атомарно, serializable, откат=no-op
-- старт: replay ops → base ; idx = reindex base ; инвариант idx ≡ reindex base (property-тест)
```

---

Суть всего: **домен — обычные типы (M:N как запись-факт, «столбцы» — любые ADT);
индексы — производный in-memory слой (`idx ≡ reindex base`); изменения — `Op`,
упакованные в монаду `Txn` (`emit` = apply+лог → `live ≡ replay`); транзакции —
serializable даром под одним писателем; разрешимость — точечно для value-инвариантов.
Persistence знает только про простые записи. Думаешь как в обычной ФП-программе.**
