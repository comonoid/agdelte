# Эволюция схемы / миграции WAL

**Статус: дизайн (2026-06-17). Продолжение [declarative-storage.md](declarative-storage.md)** —
шаг 4 того вектора, но для НАШЕГО движка (WAL + in-memory), а не Postgres. Закрывает open-end
«эволюция схемы» из [plan-crm-core.md](../plan-crm-core.md).

## 1. Проблема

WAL — append-only лог операций. Каждая операция сериализуется кодеком сущности на её **текущей**
`Schema` (`encodeRow`). Поменяли поле в записи (добавили/убрали/переставили/сменили тип) → старые
записи в `crm.wal`, записанные старым кодеком, **новым `decodeRow` не читаются** → старт-реплей падает
(`decode = nothing` → по recovery-дизайну `die`). При любом росте схемы это реальный, не отложенный
риск: один деплой со сменой поля — и история нечитаема.

## 2. Текущий формат (4 слоя, версии НЕТ)

```
байт-фрейм (FFI):     <byteLen>:<crc32>:<payload>           один record = одна транзакция
txn (Agda):           concat [ lp (encodeOp opᵢ) ]          ops через length-prefix
op (Agda):            <tag-char> <> encodeRow <schema> ...   "P"=SetParty, "p"=DelParty, …
row (Agda):           concat [ lp (encAtom (cty c) v) ]      поля в порядке колонок
```

Ключевые свойства декодера (`Agdelte.Storage.Wire`):
- `R A = List Char → Maybe (A × List Char)` — ридер по остатку входа;
- `fieldR dec` — читает один `lp`-фрейм; **пустой вход → fail**;
- `runR m s` — успех **только при полном потреблении** входа (`just (a , [])`); хвост → fail.

Из этих двух свойств следует ровно то, что Tier 1 ниже эксплуатирует и чего он НЕ может.

## 3. Модель — два уровня

### Tier 1 — аддитивная эволюция (бесплатно, без версий) ← покрывает ~80%

**Правило:** новые поля **только дописываются в КОНЕЦ** схемы и являются `CMaybe _` (как
optional-поля в Protobuf — порядок неизменен, номера только растут).

**Механика:** толерантный декод. Старая запись имеет N полей, новая схема — N+1. Декодер читает N
полей нормально; на (N+1)-м вход уже пуст → вместо fail подставляет **дефолт** (`defaultOf`), ничего
не потребляя. После этого остаток входа пуст → `runR` доволен. Новое поле у старых записей = `nothing`.

- `defaultOf : (t : ColTy) → Maybe ⟦ t ⟧` — `CMaybe _ → just nothing`; остальное → `nothing`
  (нельзя молча придумать значение обязательному полю). То есть **дописывать можно только `CMaybe`**
  (или тип с явным дефолтом — Tier 1.5, см. §7).
- `fieldOrDefaultR def dec` — при пустом входе отдаёт `def` (не потребляя), иначе `fieldR dec`.
- `decodeRowTolerant` — `decodeRow`, где `fieldR` заменён на `fieldOrDefaultR (defaultOf …)`.

**Свойства (доказуемы спайком):** полная запись декодится **идентично** строгому `decodeRow` (дефолт
не срабатывает, пока вход не пуст) → нулевой риск для существующих данных; запись на старой (N-полей)
схеме читается новой (N+1) схемой, новое поле = `nothing`.

**Чего Tier 1 НЕ умеет** (требует Tier 2): переставить/переименовать/сменить тип поля; убрать поле
из СЕРЕДИНЫ (сдвиг позиций ломает позиционный декод); убрать поле с КОНЦА (оставит хвост → `runR`
fail — нужно ослабленное потребление). Дроп — редок; при необходимости — через Tier 2 или осознанное
ослабление `runR`.

### Tier 2 — версионированная миграция (для ломающих изменений)

Когда аддитивности мало (переставить/сменить тип/переименовать/выкинуть из середины):

1. **Версия стора** `storeVer : ℕ` — растёт при любом ломающем изменении любой сущности.
2. **Версия в записи.** Перед payload транзакции пишется маркер версии: `Vn|<payload>`. Записи без
   маркера = версия 0 (легаси, текущий формат) — определяется по отсутствию префикса (sniff на старте
   payload). Так загрузка совместима снизу вверх без разовой конвертации файла.
3. **Хранятся ВСЕ исторические схемы** + миграции между соседними версиями.
4. **Реплей = ленивый upcast.** Запись декодится схемой своей версии → прогоняется через цепочку
   миграций v_k→v_{k+1}→…→current → применяется. WAL **никогда не переписывается** (append-only
   чистота сохранена, как и в no-snapshot дизайне).

**Миграция = ещё один интерпретатор над `Schema`.** Структурную часть выводим из ДИФФА схем по имени
колонки — `migrateRow : (from to : Schema) → Row from → Maybe (Row to)`:
- колонка `to` есть в `from`, тип совпал → копируем значение;
- колонки `to` нет в `from` → `defaultOf` (как Tier 1);
- колонка `from` нет в `to` → отбрасываем;
- тип не совпал / переименование / split → **custom upcast** (escape hatch: `Row from → Maybe (Row to)`,
  пишется руками на эту пару версий).

Это ровно «опиши структуру → выведи поведение», как кодек/индексы/DDL: тот же `Schema` — четвёртый
интерпретатор (миграция).

## 4. Дисциплина (инвариант разработки)

- **Порядок колонок — append-only.** Новые поля только в конец. Не переставлять, не вставлять в
  середину без Tier 2 + bump версии.
- **Новое поле — `CMaybe`** (или с явным дефолтом, Tier 1.5).
- Ломающее изменение ⇒ bump `storeVer` + миграция + тест на реплей старого формата.

## 5. Связь с реактивным / declarative-storage

Это шаг 4 вектора: схема — единственный источник для кодека, индексов, DDL, **и теперь миграции**.
partial-RAM/Postgres-эволюции по-прежнему = смена интерпретатора, не переписывание. Tier 2 ленивый
upcast = event-sourcing upcaster, но выводимый из дифф-а схем, а не руками на каждый случай.

## 6. План раскатки (инкременты)

1. **Tier 1 core** (этот заход): `defaultOf` + `fieldOrDefaultR` + `decodeRowTolerant` в
   `Agdelte.Storage.Schema` + спайк-тесты (полная запись ≡ строгий декод; старая запись читается новой
   схемой). Само по себе, живой стор не трогает.
2. **Перевести сущности на толерантный декод** (как делали миграцию кодека — по одной, под round-trip).
   После этого «добавить `CMaybe`-поле в конец» работает без миграций.
3. **Доказать на реальной сущности**: добавить поле в конец, прогнать старый WAL → читается.
4. **Tier 2** (когда понадобится ломающее изменение): версия в записи + `migrateRow` структурный +
   escape hatch + ленивый upcast при реплее.

## 7. Открытые вопросы / не-цели

- **Tier 1.5 — дефолт для не-`CMaybe`:** позволить дописывать не-nullable поле с явным дефолтом
  (список `(имя, дефолт)` рядом со схемой). Просто, но пока YAGNI — начинаем с `CMaybe`-only.
- **Дроп с конца** ослаблением `runR` (разрешить хвост) — отдельное решение; пока через Tier 2.
- **Где хранить `storeVer`** и таблицу миграций — у стора (CRM) или генерик в `agdelte-store`. Tier 2.
- **Снапшоты/ротация** ([plan open-end](../plan-crm-core.md)) — ортогонально: ротация может ЗАОДНО
  поднимать версию (rewrite на текущую), но это компакция, не обязательная для эволюции.
- **Не-цель:** автоматический вывод custom upcast (переименование/тип) — он по определению ручной.
