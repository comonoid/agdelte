# Концепция: декларативное хранилище («реактивный стор»)

> Статус: **дизайн-вектор** (пишется до реализации). Дополняет
> [storage-model.md](storage-model.md) (текущий WAL+in-memory движок) и
> [adr/0001](../adr/0001-storage-wal-in-memory.md). Реализуется инкрементально;
> SQL **не нужен сейчас**, но слой проектируется так, чтобы и SQL, и масштабирование
> легли как смена интерпретатора, а не переписывание.

## Идея в одной фразе

**Хранилище устроено как реактивный UI: ты декларативно ОПИСЫВАЕШЬ структуру данных
(«схему-template»), а вся операционка — кодек, индексы, SQL, пейджинг — ВЫВОДИТСЯ
интерпретаторами над этим описанием.** Один фундаментальный паттерн фреймворка —
«реифицируй структуру, выведи поведение» — применённый к данным так же, как он уже
применён к UI.

## 1. Почему это та же архитектура, что реактивный UI

Реактивный слой (`src/Agdelte/Reactive/Node`) построен на сдвиге: вместо
`view : Model → Html` (функция, перестраивающая дерево) — `template : Node Model Msg`
(**структура данных** с биндингами). Рантайм один раз обходит `Node` и **выводит** всю
машинерию: создание DOM, регистрацию биндингов, отслеживание изменений. Несколько
рендер-таргетов (HTML / SVG / WebGL) — это разные **интерпретаторы одного `Node`**.

Хранилище зеркально:

| Реактивный UI | Хранилище |
|---|---|
| `template : Node Model Msg` (данные, не функция) | `schema : Schema E` (данные) |
| `update : Msg → Model → Model` (чистый переход) | `apply : Op → Base → Base` (чистый переход) |
| рантайм интерпретирует `Node` → DOM + биндинги | интерпретаторы выводят `Schema` → кодек / индексы / DDL / CRUD / пейджинг |
| `Binding { extract ; equals }` — «когда обновить узел» | экстрактор индекса `E → List ℕ` — «по чему индексировать» |
| `foreach (Model→List A) render` — динамич. список | дочернее отношение 1:N (коллекция через FK) |
| `when (Model→Bool)` — условный узел | `Maybe` / nullable-колонка |
| HTML / SVG / WebGL — рендер-таргеты над `Node` | in-memory / WAL-кодек / SQL / disk-paged — таргеты над `Schema` |

Соответствие точное: `Model↔Base`, `Msg↔Op`, `update↔apply`, `template↔schema`,
рендер-таргет↔storage-backend. **Это не два механизма, а один.**

## 2. Семя уже посажено — в индексах

Декларативную деривацию стор **уже делает** — для вторичных индексов:

```agda
IM.empty (extActEng ∷ extPartParty ∷ [])    -- ДЕКЛАРАТИВНЫЙ список экстракторов
extActEng a = aEngagementId a ∷ []            -- это буквально Binding.extract для индекса
```

Экстрактор `E → List ℕ` ≡ реактивный `Binding.extract`: «вот по какому полю
отслеживать», а `IndexedMap` **сам выводит** поддержание индекса при insert/delete (как
рантайм сам обновляет DOM при change). Осталось распространить эту деривацию с индексов
на **кодек** и на **backend** (in-memory → диск/SQL).

## 3. Что НЕ декларативно сейчас (зазор)

Кодек сущностей (`encParty`/`decParty`/`encActivity`/…) написан **руками, позиционно**,
по одному на сущность. Это «тривиальный непрозрачный» путь: round-trippable, но структуры
ноль. Цена уже платилась — аудит ловил byte-vs-char баг рамки WAL и порядок-полей-грабли;
SQL-схема (§5.6 SPEC) пишется отдельным источником правды. Лекарство — вывести формат из
схемы, а не писать его.

## 4. Декларативное ядро: `Schema` / `ColTy`

Сущность получает один **декларативный «storage-template»** — значение, описывающее её
реляционную форму:

```agda
data ColTy : Set where
  CNat CStr CBool : ColTy
  CEnum  : ℕ → ColTy            -- конечный nullary-sum (N вариантов) → smallint
  CMaybe : ColTy → ColTy        -- nullable
  CFK    : String → ColTy       -- = CNat + ссылка на таблицу T (FK)

record Column : Set where
  field name : String ; ty : ColTy
Schema = List Column            -- первая колонка = суррогатный PK
```

**Граница безопасного подмножества зашита в `ColTy`.** Туда физически нельзя положить:
функцию (`a→b`), истинный полиморф, nested-datatype, экзистенциал/GADT с скрытым типом,
кодату. Подмножество ограничено *конструкцией типа*, а не дисциплиной — ровно как `Node`
не даёт положить в шаблон непредставимое. Это и есть «первопорядковые мономорфные
суммы-произведений над атомами, замкнутые по регулярной рекурсии» из теории: произведение
= строка, FK = реифицированный указатель, коллекция = дочернее отношение.

Сущности **остаются эргономичными Agda-записями** (домен-логика не меняется); `Schema` —
*дополнительное* значение рядом + крошечные `toRow`/`fromRow`. Эргономика ФП сохранена,
реляционная форма — явно реифицирована.

## 5. Интерпретаторы («рендеры» хранилища)

Из ОДНОГО значения `Schema E` выводятся (каждый — свёртка по схеме):

```
schema E →  codecOf   schema : E ↔ String          -- формат WAL (вместо ручных encX/decX)
            indexesOf schema : List (E → List ℕ)    -- какие вторичные индексы
            ddlOf     schema : "CREATE TABLE …"      -- Postgres-схема (когда понадобится)
            crudOf    schema : Op → "INSERT … ON CONFLICT …"
            backendOf schema : выбор материализации (см. §6)
```

«Впихнуть постгрес» = дописать `ddlOf`/`crudOf` над **той же** схемой — как добавить
SVG-рендер над тем же `Node`. Двойной ручной §5.6-маппинг исчезает: схема — единственный
источник правды для отображения сущность↔таблица.

## 6. Масштабирование: partial-RAM ≡ SQL ≡ смена backend

Сейчас (`storage-model.md` §17): **вся `Base` в RAM** (`Data.Map`), WAL — append-only лог
на диске, старт = реплей всего WAL → пересборка `Base` в RAM. Потолок RAM = весь датасет;
старт линеен по логу.

Ключевое наблюдение: **partial-RAM и Postgres — одна задача** — сменить материализующий
backend за абстрактным реляционным интерфейсом (`IndexedMap`/`Schema`). Декларативный слой
и есть та косвенность, которая делает это сменой интерпретатора. Варианты, по возрастанию:

1. **Снапшоты + ротация WAL** — периодически дампить `Base`, обрезать лог. Ограничивает
   *время старта*, но НЕ RAM (вся `Base` всё ещё в памяти). Половинчато.
2. **Disk-paged backend** — заменить in-memory `NatMap` структурой с пейджингом (LSM/B-tree
   /page-cache): горячие строки в RAM, холодные на диске. Интерфейс `IndexedMap` тот же,
   меняется **реализация**. Это и есть partial-RAM. Дорого писать хорошо, но локализовано.
3. **Делегировать реальному движку** (Postgres / SQLite / RocksDB) — `backendOf` целит во
   внешний движок, который **уже умеет пейджинг с диска**. `Schema → DDL/CRUD`. Прагматичный
   ответ: не писать пейджинг-движок, а ИСПОЛЬЗОВАТЬ готовый. Postgres/SQLite дают partial-RAM
   даром. Поэтому partial-RAM ≡ «своп материализации на настоящую БД» ≡ SQL-интерпретатор —
   **один и тот же своп.**

Сопутствующий сдвиг модели: чистый event-sourcing (WAL = истина, RAM = проекция) неудобен
для partial-RAM — нельзя «реплеить часть». Эволюция: **WAL остаётся durable-логом, а
запрашиваемое состояние материализуется в пейджинг/SQL-backend** (проекция, которую
*запрашивают*, а не держат целиком в RAM). Это CQRS / event-sourcing-with-materialized-views.
Декларативная `Schema` — то, что делает эту проекцию выводимой, а не ручной.

**Вывод для «сейчас»:** даже без SQL и без пейджинга, ввод декларативного слоя сегодня —
это фундамент, на котором оба будущих апгрейда (SQL, partial-RAM) становятся механикой.
Поэтому строить его имеет смысл до того, как припрёт масштаб.

## 7. Что сознательно НЕ делаем (фронтир, который не нужен)

Богатый фронтир из теории (Cheney/Lindley/Wadler: NRC + query shredding для вложенных
коллекций без N+1; суммы-как-сущности с payload через table-per-constructor; closure-table
/ nested-set для глубокой рекурсии) — **нам не нужен**. Мы держим сущности плоскими:
произведения атомов + FK + дочерние отношения через отдельные `IndexedMap`. Это самый
сладкий, дешёвый кусок подмножества: индексы работают, join'ы простые, set-at-a-time, нет
N+1 и стоимости обхода деревьев. Граница — осознанное упрощение, а не недоделка.

## 8. Инкрементальный путь (без big-bang)

1. **Спайк на одной сущности** (`Account` — самая маленькая): ввести `ColTy`/`Column`/
   `Schema` + `Row` + генерик `codecOf`, вывести `toRow`/`fromRow`, прогнать round-trip
   против старого `encAccount`/`decAccount` (`CrmWireTest`). Проверить эргономику HList в Agda.
2. Если ложится чисто — раскатать схему на остальные сущности (по одной, round-trip-страховка).
3. Вывести `indexesOf` из схемы (свернуть текущие экстракторы в декларацию схемы).
4. **Отдельной развилкой** (когда понадобится масштаб): выбрать backend-эволюцию — снапшоты
   → disk-paged → внешний движок — и реализовать `backendOf`/`ddlOf`/`crudOf`. Домен не трогается.

## 9. Открытые вопросы

- **Эргономика `Row`/HList в Agda** — насколько колюче типизированное представление строки
  по схеме; не убьёт ли оно простоту записей (спайк это и проверяет).
- **Event-sourced vs materialized** — оставлять WAL источником истины + проекция, или при
  переходе на внешний движок он становится истиной, а WAL — только аудит-лог? (см. §6).
- **Эволюция схемы** — версионирование `Schema` + конвертер при смене формата (уже открытый
  конец в `storage-model.md` §15; декларативная схема делает diff схем выводимым → миграции
  выводятся, а не пишутся).
- **Где живёт `Schema`** — рядом с сущностью (в `Crm.Identity`) или в генерик-слое
  (`agdelte-store`) как часть `WalConfig`? Скорее: DSL в `agdelte-store`, значения схем —
  у домена.

## 10. Резюме

Текущий стор уже на 80% в нужной форме (произведения + FK + дочерние отношения + декларативные
индексы-экстракторы). Зазор — ручной кодек и неабстрагированный backend. Декларативный слой
(`Schema` + интерпретаторы) закрывает оба, даёт фреймворку **одну мета-идею на UI и хранилище**,
и превращает будущие SQL и partial-RAM в **смену интерпретатора над неизменной схемой**. SQL не
нужен сейчас — нужен фундамент, который делает его (и масштаб) необременительным потом.
